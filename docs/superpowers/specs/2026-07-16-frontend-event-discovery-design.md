# 틈서울 프론트엔드 행사 탐색 개선 설계

## 1. 목표

`통합` 브랜치의 현재 UI 정체성은 유지하면서, 18,909건의 서울 문화행사 데이터를 실제 사용자가 탐색할 수 있도록 조회 정책, 캘린더 동기화, 검색·필터, 로딩·오류 처리, 모바일 사용성, 테스트를 개선한다.

성공 기준은 다음과 같다.

- 홈에서는 오늘 진행 중이거나 이후 예정인 행사가 우선 노출된다.
- 행사 목록은 기본적으로 진행 중·예정 행사만 보여주며, `지난 행사 포함`을 선택한 경우에만 과거 행사까지 검색한다.
- 캘린더는 현재 보고 있는 월의 날짜 범위만 API에 요청한다.
- API 요청 중 로딩, 빈 결과, Render 절전 복구, 일반 오류 상태가 구분되어 표시된다.
- 동일 조건 요청은 짧은 시간 동안 캐시되어 중복 호출을 줄인다.
- 모바일에서도 검색·필터·캘린더·행사 카드가 가로 넘침 없이 동작한다.
- 프론트엔드 단위 테스트와 프로덕션 빌드가 통과한다.

## 2. 검토한 접근법

### 접근 A: 화면별 최소 패치

각 Vue 컴포넌트에서 직접 날짜 파라미터를 추가하고 목록 개수를 늘린다.

- 장점: 변경 파일이 적다.
- 단점: 홈, 목록, 캘린더에서 조회 규칙이 다시 달라질 가능성이 높고 중복 코드가 증가한다.

### 접근 B: API 계층과 조회 상태를 모듈화 — 채택

`festivalsApi.js`에서 날짜·검색 파라미터를 일관되게 지원하고, 공통 날짜 유틸리티와 요청 캐시를 둔다. 각 화면은 자신의 역할에 맞는 기간만 전달한다.

- 장점: 기존 UI를 유지하면서 문제의 근본 원인인 조회 정책 불일치를 제거할 수 있다.
- 장점: 테스트 범위가 명확하고 백엔드 수정 없이 적용 가능하다.
- 단점: 최소 패치보다 변경 파일이 많다.

### 접근 C: Pinia 기반 전역 행사 스토어 전환

모든 행사 조회와 필터 상태를 전역 스토어에서 관리한다.

- 장점: 대규모 확장에는 유리하다.
- 단점: 현재 범위에는 과도하며 기존 구조를 크게 바꾼다.

## 3. 아키텍처

```text
Vue 화면/컴포넌트
  ├─ 홈 진행·예정 행사 섹션
  ├─ 월간 캘린더
  └─ 행사 목록/검색
          ↓
행사 조회 서비스
  ├─ 쿼리 파라미터 직렬화
  ├─ 응답 정규화
  ├─ 짧은 TTL 캐시
  └─ AbortSignal 지원
          ↓
Render FastAPI /api/festivals
```

백엔드 API 계약은 변경하지 않는다. 기존 `start_date`, `end_date`, `q`, `page`, `size` 파라미터를 사용한다.

## 4. 조회 정책

### 홈 행사 목록

- 기준일: 브라우저의 오늘 날짜
- 조회 범위: 오늘부터 90일 후까지
- 최대 18건 요청
- 진행 중 행사와 가까운 예정 행사 순으로 노출
- 카테고리 필터는 현재 UI를 유지

### 홈 캘린더

- 하드코딩된 `2026-07-18`을 제거한다.
- 최초 표시 월은 현재 월이다.
- FullCalendar의 `datesSet`에서 보이는 날짜 범위를 수신한다.
- 같은 범위를 중복 요청하지 않는다.
- 월 이동 시 해당 범위의 행사만 요청한다.

### 행사 목록

- 기본 조회: 오늘 이후 진행 중·예정 행사
- 검색어가 없을 때도 과거 행사는 제외한다.
- `지난 행사 포함` 체크 시 시작일 제한을 제거한다.
- 검색어 입력은 300ms 디바운스한다.
- 페이지 크기는 24건으로 하고 이전/다음 페이지를 제공한다.
- URL 쿼리 `q`, `page`, `includePast`와 화면 상태를 동기화해 새로고침 후에도 조건을 보존한다.

## 5. 상태 및 오류 처리

각 데이터 영역은 다음 상태를 구분한다.

- `loading`: 초기 요청 및 조건 변경 요청
- `success`: 데이터 표시
- `empty`: 정상 응답이지만 결과 없음
- `waking`: Render 무료 인스턴스가 깨어나는 것으로 추정되는 장기 요청
- `error`: 네트워크 또는 HTTP 오류

8초 이상 응답이 없으면 “서버를 깨우는 중” 안내를 표시하되 요청은 계속 유지한다. 재시도 버튼은 현재 조건으로 다시 요청한다.

요청 경합은 `AbortController`로 이전 요청을 취소해 늦게 도착한 응답이 최신 화면을 덮지 못하게 한다.

## 6. 캐시

`festivalsApi.js` 내부 메모리 캐시를 사용한다.

- 키: API 경로와 정렬된 쿼리 문자열
- TTL: 60초
- 동일 요청 진행 중에는 같은 Promise를 공유
- 오류 응답은 캐시하지 않음
- 명시적 재시도는 캐시를 우회

이 캐시는 브라우저 탭 수명 동안만 유지한다.

## 7. 모바일 및 접근성

- 검색·필터 영역은 760px 이하에서 세로 배치한다.
- 체크박스와 버튼에 명확한 레이블과 포커스 스타일을 제공한다.
- 로딩·오류 메시지에는 `aria-live`를 적용한다.
- 캘린더 컨테이너는 작은 화면에서 최소 폭 강제 대신 글자 크기와 행 높이를 조정한다.
- 행사 카드의 긴 제목과 주소는 적절히 줄바꿈한다.

## 8. 테스트

Node 내장 테스트 러너를 유지한다.

- API 쿼리 직렬화: 날짜·검색·페이지·지난 행사 옵션
- 날짜 유틸리티: 오늘, 월 범위, FullCalendar exclusive end 처리
- 캐시: 동일 요청 중복 제거, TTL 만료, 강제 재시도
- 매핑: `event_start_date`, `event_end_date`, `period` 변환
- 프로덕션 빌드: `npm run build`

Vue DOM 단위 테스트 도입은 이번 범위에서 제외한다. 추가 테스트 프레임워크 설치 없이 순수 로직을 분리해 검증한다.

- `frontend/src/views/FestivalListView.vue`

- `frontend/src/components/home/FestivalCalendarSection.vue`
- `frontend/tests/festivalsApi.test.js`

백엔드, DB 스키마, 기존 게시판·지도·챗봇 API는 변경하지 않는다.
